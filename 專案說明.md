# HTML/JS 程式碼分析工具 - 專案說明

## 專案概述

**HTML/JS 程式碼分析工具** 是一個自動化安全分析與代碼審查工具，用於檢測 LLM 生成的 HTML 檔案中的 JavaScript 程式碼風險與潛在副作用。

### 核心功能

1. **函式結構分析** - 提取並分類內嵌 JavaScript 函式，識別呼叫關係
2. **副作用檢測** - 標識涉及網路、DOM、儲存體、執行等危險操作的函式
3. **SBOM（軟體物料清單）檢查** - 掃描外部依賴並檢測已知漏洞
4. **互動式報告** - 以 Markdown、HTML、Mermaid 流程圖多種形式呈現分析結果
5. **Gradio Web 介面** - 無需安裝依賴即可上傳分析

## 技術架構

### 主要模組

#### 1. **analyzer.py** - 核心分析引擎
- **HTMLJSAnalyzer 類**
  - 使用 BeautifulSoup 解析 HTML 結構
  - 使用 tree-sitter JavaScript 解析器進行 AST 分析
  - 檢測內嵌 `<script>` 標籤中的代碼
  - 識別外部資源引用（`<script src>`, `<link rel="stylesheet">`）
  
- **函式分類**
  - **主函式**：具有副作用、呼叫其他函式或自訂函式宣告的函式
  - **UI 常規腳本**：僅包含內建方法（如 `forEach`, `addEventListener`）的匿名函式
  
- **副作用類別** (SIDE_EFFECTS_MAP)
  - `network`: fetch, XMLHttpRequest, WebSocket, sendBeacon, EventSource
  - `execution`: eval, Function, setTimeout, setInterval
  - `dom`: innerHTML, outerHTML, write, insertAdjacentHTML
  - `storage`: localStorage, sessionStorage, indexedDB, cookie
  - `sensitive`: clipboard, geolocation, getUserMedia
  - `dynamic`: import

#### 2. **sbom.py** - SBOM 與漏洞檢查（新增功能）

**核心特性**：混合線上/離線漏洞資料庫架構

##### CDN URL 解析
支援主流 CDN 的版本提取模式：
- **jsdelivr/unpkg** 格式: `@package@version` 或 `/npm/@scope/package@version`
- **cdnjs/bootstrapcdn** 格式: `/package/v1.2.3/`
- **檔名版本提取**: `jquery-3.4.1.min.js` → `jquery@3.4.1`

##### 線上漏洞資料庫 (OSV.dev)
```python
refresh_vulnerability_db(timeout=10)
```
- 在應用程式啟動時自動呼叫
- 使用 ThreadPoolExecutor 並發查詢 6 個主要前端函式庫
- 查詢的套件: `jquery`, `bootstrap`, `lodash`, `moment`, `axios`, `angular`
- 超時時間可配置（預設 10 秒）

##### 離線備援機制
若 OSV.dev 連線失敗或逾時，自動使用內建靜態漏洞資料庫：
- 涵蓋 jQuery 4 個 CVE、Bootstrap 3 個、Lodash 3 個、Moment 2 個、Axios 2 個、Angular 2 個
- 包含 CVE ID、嚴重級別、描述、修復版本資訊

##### 版本比對邏輯
- 將版本字串轉換為版本元組 (major, minor, patch)
- CVE 影響範圍: `introduced_version ≤ actual_version < fixed_version`
- 僅對精確版本進行檢測（至少 3 位數版本號）

##### 資料來源與時間戳
每次查詢後更新 `_db_metadata`：
- `source`: 顯示來源（"OSV.dev 線上資料庫" / "內建靜態清單"）
- `updated_at`: UTC 時間戳 (格式: `YYYY-MM-DD HH:MM UTC`)
- 所有報告中顯示此資訊

#### 3. **export_html.py** - HTML 報告生成

新增 SBOM 相關功能：
- **_generate_sbom_html()** - 生成 SBOM 卡片 HTML
  - 顯示套件名稱、版本、來源
  - 以卡片方式展示已知漏洞清單
  - 危險套件高亮顯示

- **SBOM 標籤頁**
  - 統計面板：外部相依套件總數、含漏洞套件數
  - 資料來源與更新時間說明
  - 每個依賴的詳細漏洞資訊

#### 4. **mermaid_gen.py** - Mermaid 流程圖生成

**Mermaid 11 相容性修正**（此會話早期修正）：
- **節點 ID 衝突解決**：
  - 舊方式: 使用淨化的函式名稱作為節點 ID
  - 新方式: 使用順序節點 ID (`n0`, `n1`, `n2`, ...)
  - 原因: 名稱如 `constructor`, `toString` 與 JavaScript Object.prototype 鍵衝突，導致解析崩潰
  
- **邊標籤引號修正**：
  - 將 `|變數: i|` 改為 `|"變數: i"|`
  - Mermaid 11 要求標籤必須用引號包圍

- **順序對應**：函式在字典中的遍歷順序決定節點 ID 分配

#### 5. **app.py** - Gradio Web 介面

修改重點：
- 啟動時自動呼叫 `sbom.refresh_vulnerability_db(timeout=10)` 刷新漏洞資料庫
- 分析函式返回 5 個輸出（新增 sbom_content）
- 新增 SBOM 標籤頁顯示相依套件分析結果

## 使用方法

### 本地運行

```bash
# 1. 安裝依賴
pip install -r requirements.txt

# 2. 啟動應用
python app.py
```

啟動後在終端中將看到：
```
SBOM vulnerability DB: OSV.dev 線上資料庫 | 更新時間: 2026-05-18 HH:MM UTC
```

### 使用流程

1. **上傳 HTML 檔案** - 點擊「上傳單檔 .html」選擇檔案
2. **開始分析** - 點擊「開始分析」按鈕
3. **檢視結果**：
   - **文字報告** - 主要函式及其副作用詳情
   - **常規腳本** - UI 輔助函式清單
   - **相依套件 (SBOM)** - 外部依賴及漏洞情況
   - **流程圖** - 函式呼叫關係的 Mermaid 圖表
4. **下載報告** - 點擊「匯出獨立 HTML 報告」獲取獨立 HTML 檔案

### 環境變數

若需要自訂 OSV.dev 查詢逾時時間，修改 `app.py` 中：
```python
_db_meta = sbom.refresh_vulnerability_db(timeout=10)  # 單位: 秒
```

## SBOM 漏洞檢查詳細說明

### 支援的套件與資料來源

| 套件名稱 | 涵蓋 CVE 數 | 漏洞類型示例 |
|---------|----------|----------|
| jQuery | 4 | XSS, 原型污染 |
| Bootstrap | 3 | XSS |
| Lodash | 3 | 原型污染, 指令注入 |
| Moment.js | 2 | ReDoS |
| Axios | 2 | ReDoS, XSRF 洩漏 |
| Angular/AngularJS | 2 | 原型污染, XSS |

### 查詢流程

```
應用啟動
    ↓
refresh_vulnerability_db(timeout=10)
    ↓
[並發] 查詢 OSV.dev API for 6 packages
    ├─ 成功 → 獲取最新 CVE 資料
    ├─ 超時/失敗 → 該套件改用靜態備援
    └─ 全部失敗 → 使用內建靜態資料庫
    ↓
生成中繼資料 (source, updated_at)
    ↓
分析用戶上傳的 HTML 中的 CDN 引用
    ↓
比對版本，標識有漏洞的依賴
```

### 報告展示

**Gradio Markdown 輸出示例**：

## 技術決策與權衡

### 為什麼選擇混合架構？

| 方案 | 優點 | 缺點 |
|-----|------|------|
| 純線上 OSV.dev | 最新資料 | 網路依賴，使用者隱私 |
| 純離線靜態資料庫 | 無延遲，100% 可用 | 資料很快過時 |
| **混合** | 兼顧新鮮度與可用性 | 邏輯複雜 ✓ 已實現 |

### 並發查詢的必要性

- 6 個套件串行查詢 = ~36-60 秒（逐個超時）
- 並發查詢 = ~10 秒（最慢的包決定）
- 使用 `ThreadPoolExecutor(max_workers=6)` 達到平衡

### 版本精確度要求

僅對 `1.2.3` 格式的版本進行漏洞匹配，忽略 `v1`, `1.2`, `1` 等不精確版本。
原因：無法確定補丁版本可能導致誤報。

## 開發紀錄

### 本會話修正

#### 1. Mermaid 圖表空白問題 (早期)
- **症狀**：無主函式時 Mermaid 圖表為空
- **修正**：添加空狀態訊息 `*無主函式呼叫關係可顯示。*`

#### 2. Mermaid 語法錯誤 (早期)
- **症狀**：`SyntaxError in text` 於 Mermaid 11
- **根本原因**：節點 ID 如 `constructor` 衝突 JavaScript prototype 鍵
- **修正**：
  - 改用順序節點 ID: `n0`, `n1`, ... 
  - 僅在顯示標籤中保留原始函式名稱
  - 邊標籤改為 `|"變數: name"|` 格式

#### 3. SBOM 漏洞檢查（新功能）
- **實現**：
  - 建立 `sbom.py` 含靜態漏洞資料庫與 OSV.dev 整合
  - 修改 `analyzer.py` 收集外部資源並分析依賴
  - 修改 `export_html.py` 生成 SBOM HTML 卡片
  - 修改 `app.py` 啟動時更新漏洞資料庫
  - 修改 `mermaid_gen.py` 修正節點 ID 衝突

#### 4. HuggingFace Spaces 部署

- 自動刷新漏洞資料庫運作正常

## 限制與已知問題

1. **SBOM 覆蓋有限** - 僅追蹤 6 個常見套件，不涵蓋所有 npm 生態
2. **版本精確度** - 需要 `x.y.z` 格式，無法處理分支版本或預發行版
3. **網路依賴** - OSV.dev 無法訪問時依賴靜態資料庫（資料可能過時）
4. **隱私考慮** - 線上查詢會向 OSV.dev 揭露使用者上傳的依賴清單
